O Veredito em 3 Frases

Onde Dominamos

Disciplina de processo. Manifestos pré-trabalho, retros estruturadas, axiomas formais, auditoria de integridade, governança multi-agent. Ninguém mais opera com esse nível de rigor.

Onde Empatamos

Estado persistente entre sessões e memória de projeto. Cursor (rules), Claude Code (CLAUDE.md + skills), Windsurf (memory), Aider (repo map) — todos têm algo. A gente faz mais profundo, eles fazem mais visual.

Onde Perdemos

Integração de IDE, indexação de codebase, autocomplete inline, UI visual, diff inteligente, app generation. Eles têm anos de polimento de UX que nossa arquitetura de texto não tem.

Critérios

As Dimensões que Avaliamos

Antes de comparar ferramentas, precisamos saber em quê. Estas são as categorias técnicas que definem qualidade de código assistido por AI em projetos de longo prazo — não só "completa código mais rápido".

P1

Disciplina de Processo

Manifests obrigatórios antes de qualquer trabalho. Fases formalizadas. Block dependencies. Retrospectivas estruturadas, não livres.

P2

Disciplina de Código

Axiomas formais (P/Q/C/S), code headers obrigatórios, file budgets, pointer integrity verificada via auditoria.

P3

Memória de Projeto

Estado persistente entre sessões. Memory layers separados (design/, project, execution). Continuidade ao retomar após pausas.

P4

Governança e Auditoria

Audit automático com múltiplos checks. Gates por block (testes, security). Override clause formal com justificativa registrada.

P5

Multi-agent e Multi-repo

Coordenação entre agents paralelos. Worktree isolation. Roles formalizados. Workflows cross-repo com Governor.

P6

Engenharia de Contexto

Slim boot HOT/WARM/COLD. Convention snippets per-block. Token budgets. Partial reads. Vocabulário AI-only denso.

P7

Segurança Estruturada

Axiomas S1-S5. Threat models obrigatórios pré-código. Security-review gates. Sensitive data protection.

P8

Integração com Ambiente

IDE integration, codebase indexing, inline completion, diff visual, terminal, browser automation, deploy.

Diferenciais

O Que Só Cognitive Architecture Tem

14 capacidades que nenhuma outra ferramenta oferece — não porque elas não saibam fazer, mas porque elas resolvem outro problema. A maioria foca em produtividade por linha de código. A gente foca em sustentabilidade de projeto.

Diferencial #1

Manifest pré-trabalho como contrato

Antes de escrever uma linha, o agent declara: o que vai fazer, por quê, em quais arquivos, quais gates precisam passar, dependências, scope explícito. Cursor e Claude Code permitem isso, mas é opcional e livre. Aqui é obrigatório e estruturado em S/M/L tiers.

Resultado: scope creep desaparece. AI não pode "ah, já que tô aqui, conserto isso também".

Diferencial #2

Retrospectivas como dados, não diário

Cada block termina com retro estruturada: o que foi construído, gates passaram, que axiomas foram violados, scope diverged, duração real. É auditável, não é prosa. 85 retros viraram base pra phase-forecast e velocity tracking.

Outras ferramentas: zero registro estruturado pós-trabalho.

Diferencial #3

Pointer integrity automático

Toda referência inter-arquivo é verificada. Se PROTOCOLS.md menciona protocols/file-reading-protocol.md e o arquivo some, audit FAIL. Não tem "broken link silencioso" — o sistema te cobra.

Nenhuma outra ferramenta verifica isso. Você descobre quebra quando o AI alucina.

Diferencial #4

Multi-agent com Governor

Múltiplos agents paralelos, cada um em sua worktree, coordenados por um Governor que integra, audita, atualiza estado. Claude Code tem sub-agents mas sem protocolo formal de hand-off. Aqui tem task packets, return packages, e board.md como SSOT.

Resultado: 4 agents podem trabalhar em paralelo sem stomp git nem race condition.

Diferencial #5

Memory layers (DX/UX/AX)

Memória separada por audiência: DX (Design — domain knowledge, arquitetura), UX (User — identidade do projeto), AX (Agent — execution state). Cada um com permissões diferentes de quem pode mudar e quando.

Cursor rules e CLAUDE.md misturam tudo num blob. Aqui é estratificado.

Diferencial #6

Axiomas P/Q/C/S com override formal

27 regras explícitas. Para violar uma, manifest precisa de axiom_override: "P5 — slim boot violated because...". Override accumulation > 5 em 30 dias = audit warning (sinal de drift).

Cursor rules são sugestões. Axiomas são contratos.

Diferencial #7

Phase-level roadmap formal

Cada fase tem exit_criteria, block_index, dependency_groups, estimated_duration, parallel_execution_plan. Phase-close exige todos os exit_criteria atingidos. Não é "tô trabalhando em algo", é "estou na phase-11 com 3/5 critérios completos".

Outras ferramentas operam block-a-block sem horizonte formal.

Diferencial #8

Convention snippets per-block

Governor lê o kind: do block (refactor / implementation / gate) e injeta SÓ os axiomas relevantes no contexto do sub-agent. Sub-agent que faz refactor não precisa saber as 5 regras de gate.

Outras ferramentas mandam o sistema inteiro toda vez.

Diferencial #9

Security axioms (S1-S5) com gates

Threat model obrigatório (S4) antes de código security-critical. Security review gate (S5) obrigatório no close de blocks com security:true. Auth chain integrity (S2), sensitive data protection (S3), untrusted input (S1).

Copilot e Cursor podem "lembrar" de pensar em segurança. Aqui é obrigatório.

Diferencial #10

Tracks (concurrent work threads)

Linhas de trabalho concorrentes com benchmark before/after por block. Permite medir se um Track de "performance do server" está realmente melhorando ou só iterando. Stagnation detection automática via benchmark delta.

Pattern só visto em pesquisa acadêmica de software engineering, nunca em ferramenta dev.

Diferencial #11

Velocity tracking que mede de fato

Coleta actual_duration_hours por tier S/M/L de cada block. Calcula velocity, forecast de phase com confidence level, detecta tendência (acelerando vs desacelerando). Não é estimativa furada — é dado.

Outras ferramentas: zero medição. Você nunca sabe se está mais rápido.

Diferencial #12

BLOCK_LOG imutável + ADRs estruturados

Histórico append-only de todos os blocks (done / forced / reverted) + ADRs formais para decisões não-óbvias. Combinação resulta em audit trail completo de "por que tomamos essa decisão e quando".

Git log conta o quê. Aqui contamos o porquê.

Diferencial #13

Sintaxe AI-only densa

Arquivos como STATE.md usam vocabulário próprio definido em _syntax.md. Compacto, denso, key:value. Boot HOT total < 4000 tokens. Outras ferramentas usam prosa que come 10x mais tokens.

Resultado: contexto sobra pra trabalho real, não pra ler arquivos de estado.

Diferencial #14

Health report composto

Um único comando gera: audit score, velocity por tier, phase progress %, design coverage %, track health. Snapshot completo do projeto em < 30s. Sem precisar abrir 8 arquivos.

Outras ferramentas: você abre 8 arquivos e tira conclusão na cabeça.

Honestidade

O Que Eles Têm Que A Gente Não Tem

Aqui não dá pra esconder. Anos de polimento de UX e features visuais colocaram essas ferramentas muito à frente em produtividade percebida. A diferença é foco: eles otimizam o ato de digitar. A gente otimiza o ato de manter projeto.

Eles ganham #1

Indexação semântica do codebase

Cursor, Cody, Continue, Aider — todos indexam o codebase com embeddings. Você pergunta "onde tá a lógica de billing" e ele acha. A gente depende do INDEX.md mantido manualmente.

Por que: requer infraestrutura (vector db, embeddings, sync). Custo alto.

Eles ganham #2

Autocomplete inline (Tab)

Copilot, Cursor, Tabnine, Windsurf — sugestão automática enquanto digita. Reduz keystrokes 30-50%. A gente não tem essa camada porque opera a nível de "block", não a nível de "linha".

Pra ter, precisaria virar plugin de IDE. É outro produto, não extensão.

Eles ganham #3

UI visual rica

Diff visual com syntax highlight, file tree, multi-file edit preview, terminal integrado, debugger. Cursor e Windsurf são editores reais. A gente é texto.

UX visual é um produto separado. Markdown só vai até certo ponto.

Eles ganham #4

Input multimodal (screenshots, designs)

Cola imagem do Figma, AI implementa. Cola screenshot de erro, AI debuga. A gente é text-only. Pra mudar, requer integração de visão, o que depende do client (Claude Code já tem; Cursor tem; nossa arch não usa).

Limitação do formato markdown puro.

Eles ganham #5

Browser automation nativa

Devin, Windsurf Cascade, Codex CLI — abrem browser, navegam, leem páginas, fazem scraping, testam aplicação web automaticamente. A gente delega isso via MCP browser tools quando disponível.

Caso de uso super específico (testing E2E, research) onde eles brilham.

Eles ganham #6

App generation from prompt

Bolt.new, v0, Replit Agent — "faz um app de delivery" e em minutos tem app deployado. A gente é o oposto: força planejamento estruturado. Diferentes audiences (prototyping vs. building).

A gente não compete nesse mercado. Eles otimizam pra demos rápidos.

Eles ganham #7

Distribuição e instalação

Cursor: baixa instalador, abre, login, funciona. A gente: clona repo, lê 12 arquivos, instala Python, configura .env, e ainda não sabe se funciona até rodar audit.sh.

Tempo do user pra primeiro valor: deles 2 min, nosso 2 horas.

Eles ganham #8

Comunidade e ecossistema

Cursor tem milhões de users, marketplace de rules. Claude Code tem skills e MCP servers comunidade. A gente tem 1 user (você). Sem rede, sem templates de terceiros, sem suporte distribuído.

Custo de produto pessoal. Vira problema se virar comercial.

Conclusão

A Real Sobre Onde Estamos

O que nós somos

Cognitive Architecture é a primeira tentativa séria de aplicar engenharia de software formal (manifests, gates, auditoria, axiomas, retrospectivas) ao trabalho com AI. Enquanto o mercado otimiza "AI escreve código mais rápido", a gente otimiza "AI escreve código que ainda fará sentido daqui a 2 anos".

O que não somos

Não somos uma IDE. Não somos um editor. Não somos um substituto pra Cursor ou Claude Code. Somos um protocolo + estrutura de governança que roda em cima dessas ferramentas. Idealmente, no futuro, somos um MCP server ou plugin que adiciona disciplina onde antes só havia velocidade.

Posicionamento de produto

O mercado tem ferramentas pra velocidade (Cursor, Copilot, Bolt) e ferramentas pra autonomia (Devin, Replit Agent). Não tem ferramenta pra sustentabilidade arquitetural de longo prazo. Essa é nossa lacuna. O cara que vai construir produto sério de 2+ anos é nosso cliente.
