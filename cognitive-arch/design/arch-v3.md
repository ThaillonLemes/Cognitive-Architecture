## 1. Context

After v2.5 closed (11 phases, 85 blocks, audit clean), a comprehensive brainstorm revealed that the architecture's core gap was **not missing features but DISCOVERABILITY** — many existing capabilities (velocity tracking, ADRs, security review, SDK Governor, pattern mining, health report) are technically present but never activated in the day-to-day workflow.

The v3 evolution focuses on **activating dormant capabilities**, **adding integrity protection**, **building knowledge mining infrastructure**, and creating a **Master Agent that conducts active use of available tools**.

## 3. Roadmap: Phases 12 through 17 (6 phases, 26 blocks)

```
Phase 12 ─ Foundation Fix              [4 blocks]   activate dormant features
Phase 13 ─ Architecture Integrity      [4 blocks]   protect critical files
Phase 14 ─ Retrospective Mining        [4 blocks]   turn retros into data
Phase 15 ─ Master Agent v1             [5 blocks]   conductor / orchestrator
Phase 16 ─ Visibility & Dashboard      [5 blocks]   HTML reporting layer
Phase 17 ─ Assertive Brainstorm v2     [4 blocks]   AI-led questionnaires
```

Total estimated duration: 6-8 weeks at historical Phase 7-11 pace.

### Sequencing rationale

1. **Phase 12 first** — without fixed velocity data and active ADRs, the rest builds on broken foundation.
2. **Phase 13 before mining/master** — protects critical files before extensive modification by upcoming phases.
3. **Phase 14 before master + brainstorm** — they consume mined patterns as input.
4. **Phase 15 (Master) before Phase 16 (Dashboard)** — Dashboard is generated by Master.
5. **Phase 16 before Phase 17** — having visual reports validates the approach before evolving brainstorm.
6. **Phase 17 last** — most sophisticated layer, depends on patterns + Master + dashboard.

## 4. File protection model (Phase 13 implementation target)

Three tiers introduced via frontmatter `protection:` field.

### Tier `immutable` — AI refuses modification without formal override

- `PROTOCOLS.md`
- `_syntax.md`
- `templates/manifest-{small,medium,large}.md`
- `templates/block-retrospective.md`
- `protocols/block-close-checklist.md`
- `protocols/pointer-integrity.md`
- `audit.sh`

Override procedure: user types confirmation phrase + writes justification line in `.governor/log.md`.

### Tier `guarded` — AI modifies with textual confirmation

- `CLAUDE.md`, `INDEX.md`
- All `protocols/*.md` (except `immutable` set above)
- All `templates/*.md` (except `immutable` set above)
- `templates/agent-roles/*.md`

### Tier `open` — Standard editing behavior

- `STATE.md`, `NEXT.md`, `board.md` (live state)
- `BLOCK_LOG.md` (append-only)
- `manifests/*.md` (frozen at block start, never re-edited)
- `blocks/*.md` (append-only retros)
- `governance/*.md` (generated outputs)
- `_brainstorm/*.md`, `_future/*.md` (scratchpads)

## 6. Brainstorm v2 principles (Phase 17 implementation target)

The current `protocols/brainstorm-pattern.md` becomes v2 with three principles:

### Principle 1 — Always recommend

Every question, regardless of subject (technical or feeling/sentiment), includes the AI's recommendation marked with rationale. User can override or accept.

### Principle 2 — Variable option count

Number of suggested options matches scenario size:
- Binary scenario → 2 options
- Standard decision → 3-4 options
- Wide design space → 5-7 options

Not artificially fitted to 1-4 or 1-5.

### Principle 3 — Open answers always permitted

User may answer with:
- Option number only (`A1=3`)
- Combined options (`A1=2+4`)
- Free text alone
- Mix of option + modifier (`A1=2 but using X instead of Y`)

AI synthesizes mixed answers correctly.

## 7. Master Agent design (Phase 15 implementation target)

### Identity

Single agent role at `agents/agent-master.md`. Read-everything, write-restricted (per Q6 decision).

### Behavior posture

Hybrid (Q5):

- **Proactive triggers:**
  - Audit not run in >2× recommended interval
  - Untreated gate failure persisting >24h
  - Block dependency unblocked but no movement
  - Security review revalidation overdue

- **Reactive behavior:**
  - Provides status when asked
  - Surfaces tool freshness when relevant to active block
  - Generates briefings on session resume after pause

### Tool freshness tracking

Maintains `governance/tools-registry.yaml` with `last_run`, `recommended_interval`, `priority` per tool. Surfaces stale items inline with user requests.

### Inter-agent communication

YAML-structured messages (D4). Format defined in block-102. Master coordinates between MMORPG-style sub-agents (server, client, launcher, protocol) when applicable; in cognitive-arch self-context, Master is the only active agent.

## 8. Deferred to future

### `_future/token-based-modes.md`

Original Q12 ("modes") concept evolved during discussion:

- Quality / Velocity distinction was thin (both produced similar outputs)
- Discovery had nothing meaningful to "disable" (already lightweight)
- The real underlying need is **economical vs full-token-spend modes** parameterized by measured token consumption
- This requires token metrics infrastructure not yet present

Deferred until token measurement is implemented (likely Phase 18 or 19 future).

## 9. Success criteria for v3

v3 is considered successful when:

1. `actual_duration_hours` filled in 80%+ of new blocks
2. At least 3 ADRs reactively created during natural work
3. Health report displayed at start of every session
4. Master Agent active and producing weekly reports
5. Dashboard HTML generated and consulted at least 1×/week
6. Brainstorm v2 used for at least 2 design decisions
7. No `immutable` file modified without override registered in log
