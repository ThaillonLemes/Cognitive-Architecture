# Phase 14 — Retrospective Mining

BRIEF: Turn 85+ written retrospectives from write-only logs into active inputs. Extract signals, detect patterns, generate `governance/patterns.md` with actionable recommendations.

## 1. Purpose

The architecture has accumulated 85+ retrospectives that nobody reads. Phase 14 builds the infrastructure to mine them automatically: extract structured signals (axioms violated, durations, scope expansions, gate failures), detect recurring patterns, and surface recommendations. The output is `governance/patterns.md` — a living document that tells the user "these are the recurring weaknesses, fix here". This is the foundation for evidence-based architecture evolution.

## 2. Goals

- A Python script reads all retros and extracts structured signals
- A pattern analyzer detects recurring issues (axioms violated 3+ times in 30 blocks, blocks of kind X averaging 2× their estimate, etc.)
- `governance/patterns.md` regenerated periodically and stays current
- Recommendations are surfaced (but never auto-applied) — user always decides

## 3. Invariants

- Retrospectives themselves remain untouched (read-only mining)
- Pattern detection is deterministic (same input → same output)
- No retro is modified or re-formatted

## 4. Dependencies

- Phase 12 complete (velocity data must be accumulating)
- Phase 13 complete (mining code in `sdk/` must respect protection)
- At least 10 retros with `actual_duration_hours` filled for meaningful velocity patterns (achievable via block-086 retroactive backfill)

## 5. Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| Pattern detection produces noise (false positives) | Med | Threshold (3 occurrences in 30 blocks per D1) tuned conservatively; user can adjust |
| Mining script breaks on retros with unusual format | Med | Retro template is `immutable` (Phase 13); old retros validated before mining; failures logged not raised |
| `patterns.md` becomes overwhelming list of issues | Low | Recommendations prioritized; only top 5 shown by default |

## 6. Validation

- Unit tests on `sdk/pattern_mining.py` covering 5+ pattern types
- End-to-end test: run against current 85-block corpus, verify output reasonableness

## 7. Exit Criteria

1. `sdk/retro_signals.py` extracts at least 6 signal types from each retro: axioms violated, scope expansion (yes/no), duration delta vs estimate, gate failures count, kind, block tier. Tests pass on full corpus.
2. `sdk/pattern_analyzer.py` detects patterns across signals using D1 threshold (3+ in 30 blocks). Returns list of `Pattern(name, evidence, recommendation)`.
3. `governance/patterns.md` exists, regenerated by `sdk/patterns_report.py`, formatted as markdown with: section per pattern type, evidence (block IDs), recommendation, last-detected timestamp.
4. `commands/pattern-mining.md` exists describing how to run the pipeline; integrated into Master Agent's tool registry (Phase 15) with 7-day default interval.
