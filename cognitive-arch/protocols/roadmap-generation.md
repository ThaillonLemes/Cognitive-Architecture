# Roadmap Generation Protocol

BRIEF: Deterministic 10-step protocol for generating a project's `phases/ROADMAP.md` from scratch. Run after `protocols/readiness-gate.md` passes. Output is a filled ROADMAP.md based on `templates/roadmap.md`.

**Prerequisite:** Readiness gate must pass before invoking this protocol (see `protocols/readiness-gate.md`). If readiness gate fails, generate a questionnaire instead.

---

## Step 1 — Read all design/ files

Read every file in `design/` except `_placeholder.md`. Build a flat list of all named systems, each with:
- System name
- One-line description
- Who interacts with it (user, admin, internal)
- What other systems it depends on

If fewer than 3 systems are identified: STOP. Return to `protocols/domain-phase-bridge.md` — domain knowledge is insufficient.

---

## Step 2 — Read PROJECT.md for user journeys

Extract from PROJECT.md:
- The 1-paragraph project description
- Target users (specific, not "everyone")
- Key constraints (performance, security, compliance)

Identify 3–5 "user journeys": things a target user should be able to do when the software is complete. Each journey will anchor at least one phase.

---

## Step 3 — Classify each system

For each system identified in Step 1, classify it:

- **DELIVERY** — System must be built as part of the core product; goes into a Phase.
- **TRACK** — System benefits from perpetual improvement (performance, scalability); goes into a Track after delivery.
- **BOTH** — System must be built (Phase) and then improved perpetually (Track). Mark for both.
- **DEFERRED** — System is documented but not required for MVP; mark as UNCOVERED + reason.

---

## Step 4 — Group DELIVERY systems into vertical slices

Apply `protocols/domain-phase-bridge.md` Steps 3–4:
- Group DELIVERY systems into coherent vertical slices.
- Each slice = the set of systems needed to deliver one user journey end-to-end.
- Order slices by dependency: what must exist before what.

Output: ordered list of slices, each with: systems, user journey it enables, prerequisites.

---

## Step 5 — Map slices to phases

Each slice becomes one or more phases. Rules:
- If a slice can be delivered in 5–12 blocks: one phase.
- If a slice requires more than 12 blocks: split into sub-phases (e.g., Phase 2A Foundation, Phase 2B Features).
- Each phase must have one "After this phase, a user can X" sentence.

Output: Phase list with: ID, name, systems covered, user-visible outcome, estimated blocks.

---

## Step 6 — Identify TRACK systems

For systems classified as TRACK or BOTH:
- Each becomes one entry in ROADMAP.md §4 Forever Tracks.
- Assign an initial benchmark target (e.g., "p99 < 10ms" for a networking system).
- Assign initial priority score 5/10 (adjust after first performance measurement).

---

## Step 7 — Build the Phase Coverage Matrix

For each phase, list which systems it builds. For each system, verify it appears in at least one phase or one Track. Systems appearing in neither are UNCOVERED gaps.

---

## Step 8 — Identify gaps and drift

**Gaps (UNCOVERED):** Systems in design/ with no Phase and no Track.
For each gap: decide — Phase, Track, or Deferred (with reason).

**Drift (potential):** Phases in any existing roadmap draft with no corresponding design/ concept.
Flag these for review: the phase may be implementing something not documented in design/.

---

## Step 9 — Score each phase with Phase Quality Rubric

Run `protocols/phase-quality-rubric.md` on each phase entry (even at the roadmap level, the user-visible outcome and exit criteria concept must be present).

Phases scoring < 6: flag and note what information is missing. Do NOT generate the phase file until the score is ≥ 6.

---

## Step 10 — Write ROADMAP.md

Fill `templates/roadmap.md` with:
- §1 Project Vision: the 1-sentence vision from Step 2.
- §2 Domain Systems: all systems from Step 1 with status (COVERED/UNCOVERED/TRACK).
- §3 Phase Coverage Matrix: table from Step 7.
- §4 Forever Tracks: list from Step 6.
- §5 Phase Backlog: phases from Step 5 not yet started.
- §6 Phase Archive: empty (no phases complete yet for a new project).
- §7 Coverage Gaps: gaps from Step 8 with resolution plan.

Save to `phases/ROADMAP.md`.

---

## After generation

- Run `commands/roadmap-audit.md` to validate the generated ROADMAP.md.
- If audit finds new gaps: go back to Step 8.
- If audit passes: ROADMAP.md is accepted.
- From this point: any new phase is generated by updating ROADMAP.md first, then calling `protocols/phase-generation.md`.

---

## Out of scope

- Generating individual phase files (that is `protocols/phase-generation.md`).
- Defining Track benchmarks in detail (that is `protocols/track-generation.md`).
- Proposing changes to an existing ROADMAP.md (that is `commands/roadmap-refresh.md`).

End of roadmap-generation.md.
