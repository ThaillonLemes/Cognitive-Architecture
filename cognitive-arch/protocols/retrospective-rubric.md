# Retrospective Rubric

BRIEF: 5 semantic criteria for evaluating the quality of a block retrospective. A retro that only says "block done, gates passed" is not useful. These criteria ensure each retro captures real learning, not just confirmation of completion.

**When to run:** During Phase C (Completion) of the SPARC execution protocol, after writing the retrospective draft and before closing the block.

**Implemented in:** `sdk/return_validator.py` → `validate_rubric()` — returns `[RUBRIC WARNING]`-prefixed strings for each failed criterion. Warnings do not block block-close; they signal a shallow retro.

---

## Criteria

### Criterion 1 — Deviation from plan

The retrospective must mention at least one thing that went differently than planned.

**Why:** If everything went exactly as planned, either the plan was too vague to notice deviation, or the block was too simple to need a retro. Every non-trivial block encounters at least one surprise.

**Detectable pattern:** Text contains any of: "actually", "instead", "unexpected", "turned out", "different from", "not as expected", "changed", "had to".

**Warning if absent:** `[RUBRIC WARNING] criterion-1: no deviation from plan mentioned — retro may be too generic`

---

### Criterion 2 — Gate result reference

The retrospective must reference the block's actual gate results (pass or fail).

**Why:** Gates are the evidence contract of a block. A retro that does not acknowledge gate results is disconnected from the block's definition of done.

**Detectable pattern:** Text contains any of: "gate", "passed", "failed", "exit 0", "pytest", "test".

**Warning if absent:** `[RUBRIC WARNING] criterion-2: no gate result referenced — confirm gates actually ran`

---

### Criterion 3 — Fix explanation (when a gate failed)

If the retrospective mentions a failure or fix, it must explain what specifically was changed to resolve it.

**Why:** "Fixed a bug" is not useful. "The regex did not handle empty frontmatter — changed to require at least one newline between delimiters" is.

**Detectable pattern:** If text contains "fail" or "fix" or "error" or "bug", it must ALSO contain at least one of: "because", "changed", "replaced", "added", "removed", "switched", "updated", "renamed".

**Warning if pattern violated:** `[RUBRIC WARNING] criterion-3: failure/fix mentioned but no explanation of what changed`

---

### Criterion 4 — "Next time" lesson

The retrospective must capture at least one forward-looking lesson: something to do differently next time, or something confirmed as correct to repeat.

**Why:** The retro's primary value is accumulated learning. Without a lesson, it is just a status report.

**Detectable pattern:** Text contains any of: "next time", "next block", "future", "lesson", "learned", "will", "should", "recommend", "avoid", "repeat".

**Warning if absent:** `[RUBRIC WARNING] criterion-4: no forward-looking lesson captured`

---

### Criterion 5 — Non-zero token counts

`tok_in` and `tok_out` must both be > 0. A value of 0 indicates the field was not filled in or the block ran in a mode where tokens were not tracked.

**Why:** Token tracking is the foundation of `commands/velocity.md` and `commands/health-report.md`. Missing token data degrades project intelligence.

**Detectable pattern:** Checked programmatically against the numeric values of tok_in and tok_out.

**Warning if zero:** `[RUBRIC WARNING] criterion-5: tok_in or tok_out is 0 — token tracking not recorded`

---

## Scoring

Rubric warnings are advisory, not blocking. A block CAN close with rubric warnings. However:

- **0 warnings:** Excellent retro.
- **1–2 warnings:** Acceptable; review the warnings before closing.
- **3–5 warnings:** Shallow retro; strongly consider expanding before closing.

The rubric score is included in the health report generated by `sdk/health_report.py`.

---

## Out of scope

- Blocking block-close based on rubric warnings (warnings are advisory only).
- Enforcing minimum word count (quality over quantity).
- Checking retro content for technical accuracy.
