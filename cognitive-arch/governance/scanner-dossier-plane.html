🎯 Padrões de Arquitetura de Software

Padrão Principal Detectado

Feature-Sliced Design (FSD)

A estrutura segue FSD com camadas bem definidas: features/ para features de produto, store/ para estado global (MobX), services/ para acesso à API, core/ para configuração e contextos.

🔎 Prova de Raciocínio — como chegamos a FSD
Sinais estruturais encontrados:
✅ features/ — camada de features do FSD (issues, cycles, members, modules, pages, dashboards)
✅ Separação clara entre store/ (estado), services/ (API) e features/ (UI)
✅ ce/ (Community Edition) encapsula a lógica de produto separada do routing em app/

Por que não Clean Architecture?
Não há domain/, usecases/ ou infrastructure/ explícitos. O FSD é mais adequado para React apps com routing baseado em arquivo.

🎨 Estética & Convenções de Código

⚠️ Achado Importante — snake_case no Frontend

O Plane detectou snake_case como convenção dominante para variáveis e funções — incomum para um projeto React/TypeScript. Isso ocorre porque:

  • O backend é Django (Python) — a API retorna campos em snake_case
  • Os stores MobX espelham a nomenclatura da API (project_id, created_by)
  • Tipos TypeScript das respostas da API usam snake_case por consistência

Regra de ouro: quando gerar código para o Plane, use snake_case em types que mapeiam para a API, e camelCase apenas em lógica React pura (handlers, useState).

⚡ Coexistência de Padrões — Vigente vs Legado
✅
VIGENTE — Seguir este padrão
store/  ·  hooks/  ·  power-k/
Arquivos mais recentes concentrados nos stores MobX e hooks personalizados. O power-k/ indica um sistema de comandos (Cmd+K) sendo desenvolvido ativamente. Código novo deve seguir esse padrão: MobX store → hook → componente.
⚠️
LEGADO TOLERADO — Não replicar
components/  ·  core/  ·  issues/
Arquivos mais antigos em components/ usam padrões anteriores ao FSD (componentes globais sem feature-slice). issues/ standalone sugere que features foram migradas progressivamente para ce/features/. Não criar novos arquivos em components/ — use ce/features/<feature>/components/.
🔎 Prova de Raciocínio — como detectamos vigente vs legado
Método: classificação por frequência × recência de modificação dos arquivos.

Os arquivos do top-terceiro mais recentemente modificados (clone com --depth 1 = todos recentes) concentram-se nos diretórios store/ e hooks/, indicando desenvolvimento ativo.

Os diretórios components/ e core/ aparecem mais nos dois terços "mais antigos" da distribuição (por mtime do git checkout), sugerindo que são código mais estabelecido / menos tocado.

Confidence: MÉDIO — repo clonado com --depth 1, então todos os mtimes são similares. Uma análise com histórico completo daria confidence ALTO.
